# **📝 系统日志设计规范 (Logging Specification)**

## **1. 🌟 核心设计原则**

1. 结构化 (Structured Logging)：
   - 严禁输出纯文本日志。所有日志必须为 **JSON 格式**，便于 ELK (Elasticsearch, Logstash, Kibana) 或 Loki 等日志系统自动解析和索引。
2. 零敏感数据 (Zero PII / Zero Secret)：
   - **绝对禁止**记录：用户原始健康数据 (`weight`, `height`)、私钥、完整 Proof 字符串（太长且无意义）、未哈希的 Nonce。
   - **必须记录**：数据的哈希值 (`input_hash`)、钱包地址（公开）、交易哈希 (`tx_hash`)、错误代码。
3. 可观测性 (Observability)：
   - 每条日志必须包含 **Trace ID** (链路追踪 ID)，以便串联“前端请求 -> 后端验证 -> 链上交易”的全流程。
4. 分级明确 (Log Levels)：
   - 严格区分 `DEBUG`, `INFO`, `WARN`, `ERROR`, `FATAL`，生产环境通常只开启 `INFO` 及以上。

------

## **2. 🏗️ 日志数据结构标准 (JSON Schema)**

每条日志必须包含以下标准字段：

```
{
  "timestamp": "2026-03-12T09:39:00.123Z",    // ISO 8601 格式，带毫秒
  "level": "INFO",                            // 日志级别
  "service": "health-zk-verifier",            // 服务名称
  "version": "v1.0.0",                        // 服务版本
  "environment": "production",                // 环境 (dev/staging/prod)
  "trace_id": "a1b2c3d4-e5f6-7890...",        // 全链路唯一 ID (关键!)
  "span_id": "f9e8d7c6-b5a4...",              // 当前操作片段 ID
  "user_id": "0xAb58...1234",                 // 脱敏后的用户标识 (钱包地址)
  "action": "VERIFY_PROOF_SUBMIT",            // 动作代码 (机器可读)
  "message": "Proof submitted for local verification", // 人类可读简述
  "context": {                                // 业务上下文 (动态字段)
    "task_id": "task_889900",
    "circuit_version": "v2.1.0",
    "input_hash": "0x9988...",                // 输入数据的哈希 (用于审计而非还原)
    "nonce_hash": "0x7766...",                // Nonce 的哈希
    "duration_ms": 45                         // 耗时
  },
  "error": null                               // 若有错误，包含 stack trace 和 code
}
```

## **3. 📊 日志级别定义与场景 (Level Definitions)**

表格

| 级别      | 颜色 | 定义                                   | 典型场景                                                     | 存储策略                       |
| :-------- | :--- | :------------------------------------- | :----------------------------------------------------------- | :----------------------------- |
| **DEBUG** | 🔵 蓝 | 详细的技术调试信息，包含变量内部状态。 | 电路加载细节、内存使用情况、中间计算步骤。                   | 仅开发环境开启，生产环境关闭。 |
| **INFO**  | 🟢 绿 | 系统正常运行的关键里程碑。             | 用户登录成功、Proof 提交接收、预验证通过、交易已发送。       | 保留 30 天 (热存储)。          |
| **WARN**  | 🟡 黄 | 非致命异常，系统自动恢复或需关注。     | 链上 Gas 费过高自动重试、Nonce 即将过期、第三方 API 延迟。   | 保留 90 天。                   |
| **ERROR** | 🔴 红 | 操作失败，需要人工介入或报警。         | ZK 验证数学失败、私钥读取失败、数据库连接断开、合约 Revert。 | 保留 1 年 (冷存储)，触发告警。 |
| **FATAL** | 🟣 紫 | 系统崩溃，无法继续运行。               | 内存溢出 (OOM)、磁盘满、核心依赖不可用。                     | 永久保留，立即电话通知。       |

## **4. 🛡️ 安全与合规特别设计 (Security & Compliance)**

鉴于医疗和金融属性，日志必须满足 **GDPR** 和 **HIPAA** 要求。

### **4.1 数据脱敏规则 (Masking Rules)**

在写入日志前，必须经过过滤器处理：

- 健康数据：❌ 禁止记录 

  ```
  {"weight": 70, "height": 175}
  ```

  - ✅ 替换为：`"input_data_hash": "0xsha256(...)"`。

- 零知识证明：❌ 禁止记录完整的 

  ```
  proof
  ```

   对象 (几 KB 太大且无审计价值)。

  - ✅ 替换为：`"proof_hash": "0xsha256(proof_string)"`。

- **IP 地址**：✅ 记录但需匿名化 (如保留前三段 `192.168.1.*`)，除非用于安全风控。

- **私钥/助记词**：🚫 **绝对禁止**出现在任何日志中（代码层应直接阻断）。

### **4.2 审计日志 (Audit Trail)**

对于关键操作（如注册机构、升级合约、验证通过），必须生成独立的 **Audit Log**，确保不可篡改。

- **内容**：Who (地址), What (动作), When (时间), Result (结果), Evidence (交易哈希)。
- **存储**：建议将关键 Audit Log 的哈希也上链，或使用 WORM (Write Once Read Many) 存储。

## **5. 💻 NestJS 实现方案 (Code Implementation)**

推荐使用 `pino` (高性能) 或 `winston` (生态丰富) 配合 `nestjs-pino`。以下以 **Pino** 为例。

### **5.1 安装依赖**

bash

```
npm install pino pino-http nestjs-pino express-request-id
```

### **5.2 配置模块 (**`app.module.ts`**)**

```
import { Module } from '@nestjs/common';
import { LoggerModule, Params } from 'nestjs-pino';
import { genReqId } from 'express-request-id';

const loggerParams: Params = {
  pinoHttp: {
    customProps: (req) => ({
      context: 'HTTP',
      user_address: req.headers['x-wallet-address'] || 'anonymous',
    }),
    transport: {
      target: 'pino-pretty', // 开发环境美化输出
      options: {
        singleLine: true,
        colorize: true,
      },
    },
    // 生产环境配置 (JSON 格式)
    formatters: {
      level: (label) => ({ level: label.toUpperCase() }),
      bindings: (bindings) => ({
        service: 'health-zk-verifier',
        version: process.env.npm_package_version || 'v1.0.0',
        environment: process.env.NODE_ENV || 'development',
        ...bindings,
      }),
    },
    redact: {
      // 自动脱敏字段
      paths: ['req.body.proof', 'req.body.privateKey', 'req.headers.authorization'],
      censor: '**REDACTED**',
    },
    genReqId: genReqId, // 自动生成 trace_id (req.id)
  },
};

@Module({
  imports: [LoggerModule.forRoot(loggerParams)],
  // ...
})
export class AppModule {}
```

### **5.3 业务代码中的日志使用 (**`verify.service.ts`**)**

```
import { Injectable, Logger } from '@nestjs/common';
import { PinoLogger } from 'nestjs-pino';
import { groth16 } from 'snarkjs';
import { createHash } from 'crypto';

@Injectable()
export class VerifyService {
  private readonly logger = new Logger(VerifyService.name);

  constructor(private readonly pinoLogger: PinoLogger) {}

  async submitProof(proofDto: any, traceId: string, userAddress: string) {
    const { proof, publicInputs, nonce } = proofDto;
    
    // 生成数据指纹 (用于审计，不存原始数据)
    const inputHash = createHash('sha256').update(JSON.stringify(publicInputs)).digest('hex');
    const proofHash = createHash('sha256').update(JSON.stringify(proof)).digest('hex');

    // 1. INFO: 接收请求
    this.pinoLogger.info({
      action: 'VERIFY_PROOF_RECEIVE',
      context: {
        input_hash: inputHash,
        proof_hash: proofHash,
        nonce_hash: createHash('sha256').update(nonce).digest('hex'),
      },
      message: 'Received ZK proof submission',
    }, 'Proof submission received');

    const startTime = Date.now();

    try {
      // 2. DEBUG: 开始验证 (生产环境此条不会显示)
      this.pinoLogger.debug({ action: 'VERIFY_START' }, 'Starting SnarkJS verification');

      const isValid = await groth16.verify(this.vk, publicInputs, proof);
      
      const duration = Date.now() - startTime;

      if (!isValid) {
        // 3. WARN: 验证失败
        this.pinoLogger.warn({
          action: 'VERIFY_MATH_FAILED',
          context: { duration_ms: duration, input_hash: inputHash },
          message: 'ZK Proof mathematical verification failed',
        }, 'Invalid Proof');
        
        throw new Error('Invalid Proof');
      }

      // 4. INFO: 验证成功
      this.pinoLogger.info({
        action: 'VERIFY_MATH_SUCCESS',
        context: { 
          duration_ms: duration, 
          input_hash: inputHash,
          circuit_version: 'v2.1.0'
        },
        message: 'Local verification passed, queuing for blockchain',
      }, 'Verification successful');

      return { success: true, hash: inputHash };

    } catch (error) {
      // 5. ERROR: 异常捕获
      this.pinoLogger.error({
        action: 'VERIFY_SYSTEM_ERROR',
        context: { input_hash: inputHash },
        error: error, // Pino 会自动序列化 error 对象 (stack, message)
        message: 'Unexpected error during verification',
      }, 'Verification process failed');
      
      throw error;
    }
  }
}
```

## **6. 🔍 日志查询与分析场景 (Use Cases)**

设计良好的日志能让您轻松回答以下问题：

### **场景 A：用户投诉“我的证明一直验证失败”**

- **操作**：在 Kibana/Loki 中搜索 `trace_id = "用户提供的ID"` 或 `user_id = "0x..."`。
- 结果：您可以看到完整的链路：
  1. `INFO`: 接收请求 (Input Hash: 0x123)
  2. `WARN`: 验证数学失败 (原因：PublicInputs 长度不匹配)
  3. `INFO`: 返回 400 错误
- **价值**：无需让用户截图，直接定位是前端构造数据错了，还是后端逻辑变了。

### **场景 B：安全审计“谁在昨天下午修改了白名单？”**

- **操作**：搜索 `action = "ISSUER_REGISTER"` AND `timestamp > "2026-03-11"`。
- **结果**：列出所有注册操作，包含操作者地址、IP、结果和交易哈希。
- **价值**：满足合规性要求，责任到人。

### **场景 C：性能优化“为什么最近验证变慢了？”**

- **操作**：聚合查询 `action = "VERIFY_MATH_SUCCESS"` 的 `context.duration_ms`。
- **结果**：生成 P95/P99 延迟图表。如果发现 P99 突然飙升，可下钻查看当时的 `ERROR` 日志或 `WARN` (如 GC 频繁)。

## **7. 🚀 部署与运维建议**

1. 日志收集：
   - 使用 **Filebeat** 或 **Fluent Bit** 采集容器日志，发送至 **Elasticsearch** 或 **Loki**。
   - 不要将日志直接写在容器本地磁盘，防止容器重启丢失。
2. 告警配置 (Alerting)：
   - **规则 1**：`ERROR` 日志数量在 5 分钟内超过 10 条 -> 发送 Slack/邮件通知。
   - **规则 2**：`VERIFY_MATH_FAILED` 比例超过 20% -> 可能电路版本不匹配或遭受攻击，立即通知。
   - **规则 3**：`FATAL` 日志 -> 电话呼叫 On-call 工程师。
3. 日志轮转 (Rotation)：
   - 本地文件日志需配置 `logrotate`，按大小 (100MB) 或时间 (每天) 切割，保留最近 7 天。
4. 成本管控：
   - 对于高频 `DEBUG` 日志，在 production 环境务必通过配置关闭，否则存储成本会爆炸。

这套日志设计方案不仅符合行业标准，还特别针对您的 ZK 系统做了**隐私保护**和**审计增强**，是毕业设计中体现“工程化能力”和“安全意识”的绝佳加分项。

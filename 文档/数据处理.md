## **1. 概述 (Overview)**

### **1.1 文档目的**

本文档详细定义了系统中敏感健康数据的全生命周期处理流程、安全架构、加密机制及信任模型。旨在确保系统在实现业务逻辑验证（如健康核保）的同时，严格保障用户隐私数据的**机密性**、**完整性**和**来源真实性**。

### **1.2 核心设计原则**

1. **数据最小化** (Data Minimization)：仅收集和处理完成业务所必需的最少数据。
2. **本地化处理** (Local-First)：所有敏感明文数据的计算均在用户端（浏览器/本地设备）完成，永不上传至服务器。
3. **零知识隐私** (Zero-Knowledge Privacy)：通过数学证明替代原始数据交换，验证方无法反推用户数据。
4. **可信源锚定** (Trusted Source Anchoring)：引入权威机构数字签名，杜绝用户伪造输入数据（解决 Oracle 问题）。
5. **双重授权** (Dual Authorization)：结合机构签名与用户签名，确保数据来源可信且操作经用户授权。

## **3. 数据全生命周期处理流程 (Data Lifecycle)**

### **3.1 阶段一：数据生成与机构签名 (Data Issuance)**

- **执行主体**：可信数据签发方（如医院信息系统 HIS）。
- **输入**：用户体检原始数据 D={h,w,bp,...}*D*={*h*,*w*,*b**p*,...} 。
- 处理逻辑：
  1. 机构使用其离线保存的私钥 SKinst*S**K**in**s**t* 对数据哈希进行签名。
  2. 签名算法：**EdDSA** (Ed25519) (兼顾安全性与电路验证效率)。
  3. 生成签名包： Package={D,Siginst,Timestamp,IssuerID}*P**a**c**ka**g**e*={*D*,*S**i**g**in**s**t*,*T**im**es**t**am**p*,*I**ss**u**er**I**D*} 。
- **输出**：将 Package*P**a**c**ka**g**e* 通过安全通道（如 QR 码、加密 API）交付给用户。
- **安全特性**：任何对 D*D* 的篡改都会导致 Siginst*S**i**g**in**s**t* 验证失败。

### **3.2 阶段二：本地接收与双重验证 (Local Reception & Verification)**

- **执行主体**：用户浏览器 (Client-side WASM)。
- **输入**：机构签名包 Package*P**a**c**ka**g**e* 。
- 处理逻辑：
  1. **机构签名验证**：前端使用预置的机构公钥 PKinst*P**K**in**s**t* 验证 Siginst*S**i**g**in**s**t* 。若失败，终止流程并报错“数据来源不可信”。
  2. **用户展示与确认**：向用户展示解密后的数据 D*D* ，请求用户确认。
  3. **用户签名生成**：用户确认后，前端调用钱包接口，使用用户私钥 SKuser*S**K**u**ser* 对 D*D* 进行二次签名，生成 Siguser*S**i**g**u**ser* 。
- **输出**：本地内存中构建完整的证明输入集合 {D,Siginst,Siguser}{*D*,*S**i**g**in**s**t*,*S**i**g**u**ser*} 。
- **安全特性**：确保数据未被传输链路篡改，且用户明确授权使用该数据。

### **3.3 阶段三：零知识证明生成 (ZK Proof Generation)**

- **执行主体**：本地 Circom WASM 引擎。
- 输入：
  - **私有输入** (Private)： D,Siginst,Siguser*D*,*S**i**g**in**s**t*,*S**i**g**u**ser* 。
  - **公共输入** (Public)： Hash(PKinst),Addressuser*H**a**s**h*(*P**K**in**s**t*),*A**dd**res**s**u**ser* 。
- 电路逻辑 (Circuit Logic)：
  1. **约束 1** (机构验签)：在电路内运行 EdDSA 验证组件，确认 Siginst*S**i**g**in**s**t* 合法且对应 Hash(PKinst)*H**a**s**h*(*P**K**in**s**t*) 。
  2. **约束 2** (用户验签)：在电路内运行 EdDSA 验证组件，确认 Siguser*S**i**g**u**ser* 合法且对应 Addressuser*A**dd**res**s**u**ser* 。
  3. **约束 3** (业务逻辑)：基于 D*D* 计算业务指标（如 BMI=w/h2*BM**I*=*w*/*h*2 ），判断是否满足条件（如 BMI<24*BM**I*<24 ）。
  4. **输出信号**：仅当上述三个约束全部满足时，输出 Result=1*R**es**u**lt*=1 。
- **产出**：生成零知识证明 π*π* (Proof) 和公共信号 PublicInputs*P**u**b**l**i**c**I**n**p**u**t**s* 。
- **清理**：计算完成后，立即从内存中擦除 D,Siginst,Siguser*D*,*S**i**g**in**s**t*,*S**i**g**u**ser* 等明文数据。

### **3.4 阶段四：链上验证与存证 (On-Chain Verification)**

- **执行主体**：Layer 2 智能合约。
- **输入**：交易负载 {π,PublicInputs}{*π*,*P**u**b**l**i**c**I**n**p**u**t**s*} 。
- 处理逻辑：
  1. **ZK 验证**：调用验证器合约，使用全局验证密钥 VK*V**K* 验证 π*π* 的有效性。
  2. **白名单检查**：检查 PublicInputs*P**u**b**l**i**c**I**n**p**u**t**s* 中的 Hash(PKinst)*H**a**s**h*(*P**K**in**s**t*) 是否存在于合约维护的 `TrustedIssuers` 映射表中。
  3. **状态更新**：若验证通过且 issuers 在白名单内，发射事件 `VerificationSuccess(user, timestamp, issuer_hash)`。
- **输出**：链上不可篡改的交易记录。

### **3.5 阶段五：业务触发 (Business Trigger)**

- **执行主体**：后端监听服务。
- **逻辑**：监听链上事件，解析出验证结果，自动执行承保、发奖等业务，**不接触任何原始健康数据**。

## **4. 数据结构定义 (Data Structures)**

### **4.1 原始数据对象 (Raw Data Object)**

```
interface HealthData {
  height_cm: number;      // 身高
  weight_kg: number;      // 体重
  blood_pressure_sys: number; // 收缩压
  blood_pressure_dia: number; // 舒张压
  timestamp: number;      // 数据生成时间戳
}
```

### **4.2 签名数据包 (Signed Payload)**

```
interface SignedPayload {
  data: HealthData;
  signature_inst: string; // 机构 EdDSA 签名 (Hex)
  issuer_id: string;      // 机构标识
  signature_user?: string; // 用户 EdDSA 签名 (Hex, 本地生成)
}
```

### **4.3 链上交易负载 (On-Chain Transaction)**

```
interface ZKTransaction {
  proof: string;          // zk-SNARK 证明 (Hex)
  public_inputs: [
    string,               // Hash(PK_inst) - 用于白名单校验
    string,               // User_Address - 用于归属校验
    number                // Result (1=Pass, 0=Fail)
  ];
}
```

## **5. 安全威胁建模与防御 (Threat Model & Mitigation)**

| 威胁场景       | 攻击向量                       | 防御机制 (本方案)                                            |
| :------------- | :----------------------------- | :----------------------------------------------------------- |
| **数据伪造**   | 用户手动修改输入数据以通过审核 | **机构数字签名**：电路内强制验证机构签名，修改数据会导致签名验证失败。 |
| **重放攻击**   | 攻击者截获旧的合法证明重复提交 | **时间戳约束**：电路内检查数据时间戳是否在有效期内；链上检查 Nonce。 |
| **中间人篡改** | 网络传输中篡改数据包           | **双重签名**：任何比特位的修改都会破坏 EdDSA 签名，导致本地或链上验证失败。 |
| **隐私泄露**   | 验证方试图从 Proof 反推数据    | **零知识特性**：数学上保证不可逆；且链上仅存储哈希和结果。   |
| **恶意机构**   | 机构签发虚假数据               | **声誉机制与白名单治理**：通过 DAO 或监管机构管理白名单，作恶机构将被移除并追责。 |
| **密钥泄露**   | 用户私钥丢失                   | **钱包最佳实践**：私钥由用户本地钱包（如 MetaMask）托管，系统不触碰私钥。 |

## **6. 合规性与审计 (Compliance & Auditing)**

### **6.1 GDPR / HIPAA 合规性**

- **数据主权**：用户始终拥有数据控制权，可随时撤销授权。
- **去标识化**：系统中不存在“用户名 + 健康数据”的关联表，天然符合去标识化要求。
- **被遗忘权**：由于原始数据从未上链，用户只需删除本地缓存即可实现“被遗忘”，无需请求服务器删除。

### **6.2 审计追踪**

- **审计线索**：所有验证结果均锚定在区块链 TxHash 上。
- **追溯机制**：若发生纠纷，审计方可要求用户出示原始的 SignedPayload*S**i**g**n**e**d**P**a**y**l**o**a**d* ，利用公开的机构公钥和链上记录进行离线复核，而无需暴露给公众。

## **7. 附录：关键技术参数**

- **证明系统**：Groth16 (平衡证明大小与验证成本) 或 PLONK (通用性更强)。
- **签名算法**：EdDSA (Ed25519)，曲线为 Curve25519。
- **哈希算法**：Poseidon Hash (针对 ZK 电路优化的哈希算法)。
- **编译工具链**：Circom v2+, SnarkJS。
- **区块链网络**：Polygon PoS 或 Arbitrum One (低 Gas 费，高吞吐量)。

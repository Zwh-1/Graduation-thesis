### **1. 🔑 密钥管理模块 (Key Management System, KMS)**

**目标**：确保可信机构私钥绝对安全，公钥分发可信且可更新。

#### ***\*1.1 架构设计\****

采用 **“离线根密钥 + 在线派生密钥”** 的分层架构，结合 **链上注册表** 进行公钥管理。

- Level 1: 根密钥 (Root Key)
  - **存储**：冷存储（空气隔离的硬件设备，如 Ledger Nano X 或专用 HSM）。
  - **用途**：仅用于签署“机构公钥证书”，不直接签署业务数据。每年仅使用一次。
  - **生成仪式**：模拟多方计算 (MPC) 仪式，由医院院长、IT 主管、第三方审计员共同见证生成。
- Level 2: 业务签名密钥 (Signing Key)
  - **存储**：医院内部受保护的服务器（加密存储），定期轮换（如每月）。
  - **用途**：直接对用户健康数据进行 EdDSA 签名。
  - **生命周期**：短期有效，过期自动作废。

#### ***\*1.2 公钥分发与验证流程\****

1. **注册**：医院管理员使用根密钥对新的业务公钥 PKbiz*P**K**bi**z* 签名，生成证书 Cert=Sign(SKroot,PKbiz∣∣ExpiryDate)*C**er**t*=*S**i**g**n*(*S**K**roo**t*,*P**K**bi**z*∣∣*E**x**p**i**ryD**a**t**e*) 。
2. **上链**：调用智能合约 `registerIssuer(cert, pk_hash)`，将公钥哈希和过期时间写入链上白名单。
3. **前端获取**：用户端启动时，从链上拉取最新的 `TrustedIssuers` 列表，缓存到本地 IndexedDB。
4. **验证**：前端在生成证明前，先校验接收到的数据签名是否对应链上白名单中的公钥。

#### ***\*1.3 核心代码逻辑 (Solidity)\****

```
struct IssuerInfo {
    bytes32 pubKeyHash;
    uint256 expiryTime;
    bool isActive;
}

mapping(bytes32 => IssuerInfo) public trustedIssuers; // key: issuerID

function registerIssuer(bytes32 _issuerID, bytes32 _pkHash, uint256 _expiry) external onlyGovernance {
    trustedIssuers[_issuerID] = IssuerInfo(_pkHash, _expiry, true);
    emit IssuerRegistered(_issuerID, _expiry);
}

function isIssuerValid(bytes32 _issuerID, bytes32 _pkHash) public view returns (bool) {
    IssuerInfo memory info = trustedIssuers[_issuerID];
    return info.isActive && info.pubKeyHash == _pkHash && block.timestamp < info.expiryTime;
}
```

### **2. 🖥️ 前端优化模块 (Frontend Optimization)**

**目标**：解决 WASM 加载慢、计算阻塞 UI、内存溢出问题，提供流畅的用户体验。

#### ***\*2.1 核心技术策略\****

- Web Workers 隔离计算：
  - 将繁重的 `witness calculation` (生成见证) 和 `proof generation` 放入独立的 Web Worker 线程。
  - **主线程**：只负责 UI 交互、文件读取、进度条更新。
  - **Worker 线程**：加载 `.wasm`，执行 Circom 逻辑，完成后通过 `postMessage` 返回 Proof。
- 流式加载与缓存：
  - 使用 Service Worker 缓存 `.wasm` 和 `.zkey` 文件，避免重复下载。
  - 对大文件实施分块加载 (Chunked Loading)。
- 内存优化：
  - 在 Witness 生成后，立即释放原始数据占用的 `Uint8Array` 内存。
  - 限制并发任务数，防止移动端崩溃。

#### ***\*2.2 用户体验流程设计\****

1. **上传/扫描**：用户上传医院签名的 JSON 或扫描二维码。
2. **预检 (Pre-flight)**：JS 快速验证签名格式和有效期（不调用 WASM），失败立即报错。
3. 计算中：
   - 显示动态进度条（基于 Circom 的 `step` 回调）。
   - 提示文案：“正在本地加密计算，数据不会离开您的设备...”。
4. **完成**：展示 Proof 哈希，提供“发送到链上”按钮。

#### ***\*2.3 代码示例 (Worker 调用)\****

```
// main.js
const worker = new Worker('zk-worker.js');

worker.onmessage = (e) => {
    if (e.data.type === 'progress') {
        updateProgressBar(e.data.percent);
    } else if (e.data.type === 'success') {
        submitToBlockchain(e.data.proof);
    } else if (e.data.type === 'error') {
        showError(e.data.message);
    }
};

// 发送数据给 Worker
worker.postMessage({
    type: 'generate_proof',
    inputData: healthData,
    signature: institutionSig
});
```

### **3. 🛡️ 安全防护模块 (Security Hardening)**

**目标**：防御重放攻击、数据篡改、中间人攻击及侧信道泄露。

#### ***\*3.1 防重放攻击 (Replay Attack Prevention)\****

- **机制**：**时间窗口 + 一次性随机数 (Nonce)**。
- 电路约束：
  - 输入包含 `timestamp` 和 `nonce`。
  - 约束 1：`current_block_time - 1 hour < timestamp < current_block_time + 10 min` (确保数据新鲜)。
  - 约束 2：`nonce` 必须未被使用过。
- **链上检查**：合约维护一个 `usedNonces` 映射 (或使用 Bloom Filter 节省 Gas)，验证通过后标记为已使用。

#### ***\*3.2 数据完整性校验\****

- **双重签名绑定**：电路中强制验证 `Sig_Inst` 和 `Sig_User` 必须针对**完全相同**的数据哈希。防止攻击者拼接合法的机构签名和用户签名到不同的数据上。

#### ***\*3.3 隐私增强\****

- 元数据混淆：
  - 用户在提交交易时，引入随机延迟 (Random Delay)。
  - 如果可能，通过中继器 (Relayer) 提交交易，隐藏用户真实 IP 和钱包地址的关联（使用 ERC-2771 标准）。

#### ***\*3.4 电路逻辑伪代码\****

```
// 防重放逻辑
component timeCheck = LessThan(32);
timeCheck.in[0] <== current_timestamp;
timeCheck.in[1] <== input_timestamp + 3600; // 1小时窗口
// ... 连接其他约束
```

### **4. ⛓️ 合约治理模块 (On-Chain Governance)**

**目标**：实现系统的可升级性、白名单动态管理及紧急熔断。

#### ***\*4.1 代理模式 (Proxy Pattern)\****

- 设计：使用 UUPS (Universal Upgradeable Proxy Standard)

   模式。

  - **Proxy 合约**：存储状态（白名单、Nonce），转发调用。
  - **Logic 合约**：包含验证逻辑和验证密钥 (VK)。

- **优势**：当电路优化或 VK 变更时，只需部署新的 Logic 合约并更新 Proxy 指向，无需迁移用户数据或改变合约地址。

#### ***\*4.2 多签治理 (Multi-Sig Governance)\****

- **角色**：设立一个由 3-5 个可信方（如：项目方、监管机构代表、社区领袖）组成的多签钱包。
- 权限：
  - 添加/移除可信机构。
  - 升级 Logic 合约。
  - 触发紧急暂停。
- **工具**：集成 **Safe (原 Gnosis Safe)** 合约作为治理入口。

#### ***\*4.3 紧急熔断机制 (Circuit Breaker)\****

- **功能**：当发现严重漏洞时，治理方可调用 `pause()` 函数。

- **效果**：所有验证交易 revert，但允许 `unpause` 和资金提取（如有）。

- 代码片段

  ```
  modifier whenNotPaused() {
      require(!paused, "System is paused");
      _;
  }
  
  function verifyAndRecord(...) external whenNotPaused {
      // 验证逻辑
  }
  ```

### **5. 🧪 测试体系模块 (Testing Framework)**

**目标**：确保电路逻辑零错误，合约无漏洞，系统抗攻击。

#### ***\*5.1 分层测试策略\****

1. 单元测试 (Unit Test)：
   - **工具**：`circom-tester`, `chai`, `mocha`。
   - **内容**：针对每个 Circuit 组件（如 BMI 计算、签名验证）编写测试用例。
   - **边界值**：重点测试 BMI=24.0, 身高=0, 负数, 最大值溢出等情况。
2. 集成测试 (Integration Test)：
   - **工具**：`Hardhat`, `SnarkJS`。
   - **流程**：模拟完整链路：生成数据 -> 签名 -> 本地生成 Proof -> 链上验证 -> 状态变更。
3. 模糊测试 (Fuzzing)：
   - **工具**：`Echidna` (针对 Solidity), 自定义脚本生成随机输入。
   - **目的**：随机输入海量非法数据，确保电路永远不崩溃且输出正确（始终为 0 或报错）。

#### ***\*5.2 自动化 CI/CD 流水线\****

- 配置 GitHub Actions：
  - 每次 Push 自动编译电路。
  - 运行所有测试用例。
  - 静态分析 Solidity 代码 (`Slither`)。
  - 生成测试覆盖率报告。

#### ***\*5.3 形式化验证 (进阶)\****

- 使用 `zkAudit` 或手动数学推导，证明电路约束等价于业务需求文档（例如：证明 `output=1` 当且仅当 `BMI < 24` 且 `Sig` 有效）。

### **6. 📜 合规文档模块 (Compliance & Legal)**

**目标**：满足 GDPR/HIPAA 要求，明确责任边界，建立用户信任。

#### ***\*6.1 核心文档清单\****

1. 隐私政策 (Privacy Policy)：
   - 明确声明：**“原始健康数据永不离开用户设备，永不存储于任何服务器。”**
   - 解释链上存储的内容仅为“零知识证明”和“公钥哈希”，无法反推个人隐私。
2. 数据处理协议 (DPA)：
   - 定义机构、用户、平台三方的数据责任。
   - 明确机构对数据真实性负责，平台对代码逻辑负责。
3. 用户知情同意书 (Informed Consent)：
   - 在首次使用时弹出，用户需勾选同意。
   - 内容：授权范围、数据用途、风险告知。
   - **技术实现**：用户签名同意书的哈希上链，作为法律凭证。

#### ***\*6.2 合规性设计细节\****

- **数据最小化原则**：电路中只输入必要的字段（如只输入 BMI 结果所需的体重身高，不输入姓名、身份证号）。
- 被遗忘权 (Right to be Forgotten)：
  - 设计说明：由于数据仅在用户浏览器内存中存在，用户关闭页面即物理删除。链上记录为匿名哈希，不属于个人敏感信息，因此天然合规。
- 审计日志：
  - 保留系统操作日志（不含用户数据），供监管机构审查系统运行情况。

#### ***\*6.3 文档结构示例\****

> **第 3 章：数据安全与合规承诺**
> 3.1 数据流向图解（强调本地化处理）
> 3.2 法律责任界定矩阵
> 3.3 用户权利行使指南（如何删除数据、如何撤销授权）
> 3.4 应急响应预案（数据泄露假设下的处理流程，虽理论上不可能泄露明文）

# **📘 系统开发文档：基于零知识证明与轻量级规则的健康隐私系统**

**Health-ZK System: A Privacy-Preserving Health Data Verification Framework**

## **1. 项目概述 (Project Overview)**

### **1.1 项目背景**

在数字化医疗时代，用户健康数据（如诊断结果、疫苗接种记录、体检指标）的共享需求日益增长。然而，传统的数据共享模式存在严重的隐私泄露风险：用户必须向验证方（如医院、保险公司、雇主）披露原始数据才能证明合规性。本项目旨在利用**零知识证明 (Zero-Knowledge Proofs, ZKP)** 技术，构建一个去中心化的隐私保护系统，使用户能在不泄露具体健康数据的前提下，证明其满足特定的轻量级规则（如“年龄>18岁”、“核酸阴性”、“血压在正常范围内”）。

### **1.2 核心目标**

1. **隐私优先**：验证方仅获知“真/假”结论，无法反推原始健康数据。
2. **轻量级规则引擎**：支持灵活配置基于阈值、范围、逻辑运算的验证规则。
3. **端到端验证**：实现从电路编译、前端证明生成、后端/链上验证的全流程闭环。
4. **高可维护性**：采用模块化单体架构，日志系统完善，符合企业级开发标准。

### **1.3 适用场景 (毕业设计演示用例)**

- **场景 A**：用户证明“收缩压 < 140 mmHg”，无需透露具体血压值。
- **场景 B**：用户证明“已接种至少 2 剂疫苗”，无需透露接种时间或疫苗品牌。
- **场景 C**：用户证明“年龄在 18-60 岁之间”，无需透露出生日期。

## **4. 核心功能模块实现 (Core Implementation)**

### **4.1 零知识电路设计 (Circom)**

**功能**：定义隐私规则。
**示例电路** (`src/rules/age_check.circom`)：

```
template AgeCheck() {
    signal input age;       // 私有输入：用户真实年龄
    signal input min_age;   // 公开输入：最小年龄限制
    signal output out;      // 公开输出：1 (通过) 或 0 (失败)

    // 约束逻辑：如果 age >= min_age，则 out = 1
    component comparator = LessThan(); 
    comparator.in[0] <== min_age - 1;
    comparator.in[1] <== age;
    
    // 如果 comparator.out == 0 (即 age > min_age - 1)，则通过
    out <== 1 - comparator.out;
}
```

- **编译流程**：通过 `scripts/build.sh` 编译，生成 `age_check.wasm` (前端用) 和 `age_check.zkey` (验证用)，并自动提取 `Verifier.sol`。

### **4.2 前端证明生成 (Frontend Prover)**

**功能**：在用户浏览器本地生成证明，数据不出域。
**关键代码逻辑** (`features/zk-prover/useZKProver.ts`)：

1. 加载 `public/circuits/age_check.wasm` 和 `zkey` 文件。
2. 收集用户输入（如 `age: 25`）和公开参数（如 `min_age: 18`）。
3. 调用 `snarkjs.groth16.fullProve()` 生成 `proof.json` 和 `publicSignals.json`。
4. 将证明发送至后端 API。

- **隐私保障**：原始 `age` 值仅在浏览器内存中存在，生成证明后立即销毁，绝不发送给服务器。

### **4.3 后端验证与审计 (Backend Verifier)**

**功能**：接收证明，进行链下预验证或直接转发至链上，并记录审计日志。
**关键组件** (`modules/zk-verifier/zk-verifier.service.ts`)：

1. **接收请求**：解析 `proof` 和 `publicSignals`。
2. **链下预验证 (可选)**：使用 `snarkjs.groth16.verify()` 和 `.zkey` 快速验证，减轻链上压力。若失败，写入 `logs/backend/errors/zk-verify-fail.log`。
3. **链上验证**：调用合约 `verifyProof()` 方法。
4. **审计记录**：验证成功后，将交易哈希、时间戳、用户 ID 写入数据库，并记录 `logs/backend/security/audit-trail.log`。

**安全守卫** (`common/guards/ZKProofGuard`)：

- 作为 NestJS Guard，可装饰在 Controller 方法上。
- 自动拦截请求，执行验证逻辑。验证失败直接抛出 403 异常，无需业务代码重复编写验证逻辑。

### **4.4 智能合约验证 (On-Chain Verification)**

**功能**：提供不可篡改的信任锚点。
**合约逻辑** (`src/registry/HealthRegistry.sol`)：

- 引入自动生成的 `Verifier.sol`。
- 暴露 `submitProof(bytes memory proof, uint[] memory publicSignals)` 函数。
- 内部调用 `Verifier.verifyProof(...)`，若返回 `true`，则触发 `VerificationSuccess` 事件，并可更新链上状态（如标记用户为“已验证”）。

## **5. 开发与部署工作流 (Workflow)**

### **5.1 本地开发流程**

1. **环境准备**：安装 Node.js, Circom, Docker。

2. 电路编译：

   ```
cd circuits
   npm run build  # 编译电路，生成产物，日志输出至 ../logs/circuits/
npm run distribute # 自动复制 .wasm 到 frontend, .sol 到 contracts
   ```

3. 合约部署：

   ```
cd contracts
   npx hardhat node # 启动本地链
npm run deploy   # 部署合约，日志输出至 ../logs/contracts/
   ```

4. 后端启动：

   ```
cd backend
   npm run start:dev # 监听端口，日志输出至 ../logs/backend/
```
   
5. 前端启动：

   ```
cd frontend
   npm run dev # 访问 localhost:3000，构建日志输出至 ../logs/frontend/
```

### **5.2 日志监控策略**

- **开发阶段**：开发者可实时 `tail -f logs/backend/app/info.log` 观察业务流。
- **调试阶段**：若 ZK 验证失败，立即检查 `logs/backend/errors/zk-verify-fail.log` 获取详细的输入哈希和错误堆栈。
- **安全审计**：定期审查 `logs/backend/security/audit-trail.log` 以监控异常访问。
- **日志轮转**：配置 `logrotate` 或 Winston 的 `DailyRotateFile`，将超过 30 天的日志归档至 `logs/archive/` 并压缩。

## **6. 安全性与隐私保护分析 (Security Analysis)**

### **6.1 隐私保护机制**

- **数据最小化**：系统仅处理证明所需的公开信号（Public Signals），原始私有数据（Private Inputs）永远保留在用户端。
- **零知识性**：验证者无法从 Proof 中推导出任何关于 Private Inputs 的信息（基于 Groth16 的安全性假设）。
- **传输安全**：所有 API 通信强制 HTTPS，证明数据在传输过程中加密。

### **6.2 安全防护措施**

- **重放攻击防御**：在 Public Signals 中加入随机数 (Nonce) 或时间戳，确保每个证明只能使用一次。
- **输入完整性**：电路中包含对输入格式的严格约束，防止恶意构造的输入导致电路行为异常。
- **访问控制**：后端实施基于角色的访问控制 (RBAC)，敏感操作需多重验证。
- **日志脱敏**：`request-dump.log` 自动过滤掉可能的敏感字段，仅保留必要的调试信息。

## **7. 测试计划 (Testing Plan)**

### **7.1 单元测试**

- **电路测试**：使用 `circomlib` 和测试输入 (`inputs/*.json`) 验证电路逻辑的正确性（如：输入 17 岁应返回 0，输入 18 岁应返回 1）。
- **后端测试**：针对 `ZKProofGuard` 和各 Service 层编写 Jest 单元测试，模拟验证成功/失败场景。
- **合约测试**：使用 Hardhat Chai 编写测试用例，覆盖 Gas 消耗边界和验证逻辑。

### **7.2 集成测试**

- **端到端流程**：自动化脚本模拟用户从输入数据 -> 生成证明 -> 提交后端 -> 链上验证 -> 查看结果的全流程。
- **性能测试**：测量不同复杂度电路的证明生成时间（前端）和验证时间（后端/链上），确保满足“轻量级”要求（目标：生成 < 5s, 验证 < 1s）。

### **7.3 安全测试**

- **模糊测试**：向电路和后端接口发送随机/畸形数据，检验系统的鲁棒性。
- **日志审计**：验证所有关键操作是否均被正确记录到对应的日志文件中。
